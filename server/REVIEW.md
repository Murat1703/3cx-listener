# Результат проверки

## Проверено

- синтаксис всех JavaScript-файлов через `node --check`;
- установка зависимостей через `npm ci`;
- отсутствие известных уязвимостей production-зависимостей через `npm audit --omit=dev`;
- локальный сквозной mock-тест:
  - GET PCM16 8000 Hz из имитации 3CX;
  - ресемплинг 8 kHz → 24 kHz;
  - отправка `input_audio_buffer.append` в OpenAI Realtime WebSocket;
  - получение `response.output_audio.delta`;
  - ресемплинг 24 kHz → 8 kHz;
  - отправка аудио в POST stream имитации 3CX.

Результат mock-теста: успешно, 16 входных audio events и 30 400 байт отправленного обратно PCM за тестовый интервал.

## Критические исправления

1. Устранено обращение к `openAiWs` до объявления переменной.
2. Исправлен импорт `./live-transcription.js`.
3. Проверка прав 3CX выполняется по конкретному DN, а не глобальному `/callcontrol`.
4. Для GET/POST stream добавлено обновление токена после `401`.
5. Для `404/424` добавлены повторные попытки и понятная диагностика Route Point.
6. Добавлено ожидание `session.updated`; входной звук не уходит до применения аудиоконфигурации OpenAI.
7. FFmpeg переведён в low-latency режим.
8. При перебивании ответа очищается PCM-очередь и перезапускается выходной ресемплер.
9. Добавлен фильтр только входящих звонков и allowlist внешних номеров.
10. Frontend WebSocket по умолчанию слушает только localhost.

## Что нельзя подтвердить без доступа к рабочей 3CX

Реальный GET/POST media stream на PBX не запускался, потому что в переданных файлах нет действующих `THREECX_CLIENT_SECRET` и `OPENAI_API_KEY`. Команда `npm run health` проверит токен и доступ к `/callcontrol/<DN>` после заполнения `.env`.

Если реальная 3CX продолжит отдавать `424`, наиболее вероятная причина — `THREECX_EXTENSION` указывает на обычный extension, а не на Route Point/API application, которому принадлежит participant.
